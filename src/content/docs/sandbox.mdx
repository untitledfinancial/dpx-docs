## The 5-call sequence

Run these in order. Steps 1–3 require no auth — they're live oracle data available to any caller. Steps 4–5 use your sandbox key.

### 1 · Protocol discovery

```bash
curl https://stability.untitledfinancial.com/manifest
```

Returns supported corridors, contract addresses, compliance posture, and tool capabilities. This is what MCP agents call on first connection.

---

### 2 · Fee quote — $500K USD→EUR

```bash
curl "https://stability.untitledfinancial.com/quote?amountUsd=500000&hasFx=true&esgScore=75"
```

Returns a binding quote valid for 300 seconds: settlement fee, FX overlay, ESG fee, and a `quoteId`. Real oracle math — same calculation used in production.

---

### 3 · Oracle stability check

```bash
curl https://stability.untitledfinancial.com/reliability
```

Current stability score, peg deviation, and `isHealthy` flag. The oracle gates settlement automatically — this gives you advance visibility. Look for `stabilityScore ≥ 90` before proceeding.

---

### 4 · FX corridor intelligence

```bash
curl "https://intelligence.untitledfinancial.com/v1/intelligence/fx-settlement" \
  -H "X-API-Key: YOUR_DEMO_KEY"
```

Live risk assessment across 10 FX corridors. Returns regime, settlement advice, DXY position, and the best/worst corridors at this moment. Backed by FRED spot rates updated hourly.

---

### 5 · Initiate sandbox settlement

```bash
curl -X POST "https://integration.untitledfinancial.com/payments/initiate?sandbox=true" \
  -H "Authorization: Bearer YOUR_DEMO_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "amount": "500000",
    "currency": "USD",
    "creditor": {
      "name": "Acme Logistics GmbH",
      "lei": "529900T8BM49AURSDO55",
      "walletAddress": "0xe333551E18ef0471A71d7e8e761212766aa5AD4f"
    },
    "debtor": {
      "name": "Acme Corp",
      "lei": "5493001KJTIIGC8Y1R12"
    },
    "purpose": "Supplier payment Q2 2026",
    "esgScore": 75
  }'
```

Runs the full compliance preflight: Verification of Payee, AML behavioural screening, ESG scoring of the counterparty entity. Returns a settlement reference, compliance result, fee breakdown, and ESG redistribution plan. No on-chain transaction — response shape is identical to production.

## What the compliance preflight checks

Every sandbox settlement runs the same three-layer check as production:

**Verification of Payee** — Confirms the creditor entity name matches the registered wallet owner. FATF R16 / MiCA Article 83 compliant.

**AML Behavioural Screening** — Three-layer analysis: sanctions list check, z-score anomaly detection against historical transaction patterns, graph-based counterparty risk propagation. No external API calls at payment time — all inference is local.

**ESG Entity Score** — Scores the counterparty business against 6 institutional ESG data sources. Score feeds directly into the fee calculation — higher-impact activity costs more, and 100% of the ESG fee component is redistributed on-chain to verified impact programs.

## Using the Postman collection

[Download the pre-built collection →](https://integration.untitledfinancial.com/demo/postman)

1. Import into Postman
2. Open **Variables** and paste your demo key into `dpx_demo_key`
3. Run calls 1–6 in order — each response feeds into the next
4. Call 6 polls settlement status by ID — copy the `id` from call 5 into the `settlement_id` variable

## Settlement transaction lifecycle

```
QUOTED → VERIFYING → EXECUTING → CONFIRMED → REDISTRIBUTING → COMPLETE
                                      ↓
                                   FAILED → REFUNDED
```

In sandbox mode, the settlement moves through `QUOTED → VERIFYING → COMPLETE` without reaching `EXECUTING`. All states before `EXECUTING` are identical between sandbox and production.
